# burn — 仕様書 / 設計書

ワンタイム秘密共有サービス（ゼロ知識・エッジ配信）

| 項目 | 内容 |
|---|---|
| バージョン | 0.1（draft） |
| 対象読者 | 実装者・レビュアー |
| 想定基盤 | Cloudflare Workers / KV /（厳密版）Durable Objects |
| スタック | Hono（Worker）+ Web Crypto（ブラウザ） |

---

## 1. 概要

パスワードや API キーといった秘密を、**一度開いたら消えるリンク**で相手に渡すサービス。
秘密はブラウザ内で暗号化し、復号鍵は URL の `#` フラグメントに載せる。サーバー（Cloudflare）には
**鍵なしの暗号文しか届かない**ため、運営者は中身を知り得ない（ゼロ知識）。

解決したい課題は、Slack やメールに平文の秘密を貼る運用が生む**恒久的な平文の残置**。
チャット履歴・バックアップ・検索インデックス・連携 bot に平文が残り続ける状態を消す。

---

## 2. 目的 / 非目的

### 目的（Goals）
- サーバー・ログ・KV のいずれが漏洩しても、平文が復元されないこと（ゼロ知識）。
- 受け取り側がアプリ導入・アカウント登録なしに、**クリックして開くだけ**で使えること。
- 一度開いたら破棄され、再アクセスでは何も得られないこと（ワンタイム）。
- エッジ配信により世界中で低遅延・低コスト・サーバー管理ゼロで運用できること。

### 非目的（Non-goals）
- パスワードマネージャ・長期保管庫ではない（短命な受け渡し専用）。
- 悪意ある運営者（バックドア JS の能動的配信）からの防御は**範囲外**（§5.3 に明記）。
- v1 ではアカウント／組織管理・監査ログ・大容量ファイルは扱わない（拡張 §11）。

## 3. 全体アーキテクチャ

```
            ┌─────────────── ブラウザ（信頼境界の内側）───────────────┐
            │  平文                                                   │
 送信者 ──▶ │  AES-256-GCM で暗号化（Web Crypto）                     │
            │  鍵 = 乱数（URL の #fragment に格納、外に出ない）        │
            └──────────────────────────┬──────────────────────────────┘
                                        │  送るのは id + 暗号文 のみ
              ── ネットワーク境界（# 以降は送信されない）──
                                        ▼
            ┌─────────────── Cloudflare エッジ（信頼境界の外側）──────┐
            │  Worker（Hono）: 受領・保管・ワンタイム破棄              │
            │  KV: { ct, iv, salt, maxViews, views }  ← 鍵なしの blob   │
            └──────────────────────────┬──────────────────────────────┘
                                        │  受信者が id で取得 → 暗号文を返す
                                        ▼
            ┌─────────────── 受信者のブラウザ ───────────────────────┐
            │  #fragment の鍵で復号 → 表示 → サーバー側は破棄           │
            └─────────────────────────────────────────────────────────┘
```

**核となる不変条件**: 復号鍵と平文は、ネットワーク境界を**一度も越えない**。
境界を越えるのは `id`（引き出し用ハンドル）と暗号文だけ。

## 4. 暗号設計

### 4.1 鍵生成・派生

| 要素 | 仕様 |
|---|---|
| 乱数源 | `crypto.getRandomValues`（CSPRNG） |
| ベース鍵 `rawKey` | 32 byte（256 bit）乱数 |
| アルゴリズム | AES-256-GCM |
| IV | 12 byte 乱数（暗号化ごとに新規） |
| パスフレーズ派生 | PBKDF2（SHA-256, iterations = 120,000, salt = 16 byte 乱数） |

鍵導出（パスフレーズ任意）:

```
material = passphrase あり ? concat(rawKey, PBKDF2(passphrase, salt, 120k, SHA-256, 32B))
                           : rawKey
keyBytes = SHA-256(material)            // 32 byte に正規化
key      = importKey(keyBytes, "AES-GCM")
ct       = AES-GCM-encrypt(key, iv, utf8(plaintext))   // 認証タグ込み
```

- `rawKey`（=URL の鍵トークン）は**サーバーに送られない**。
- パスフレーズありの場合、鍵は「`rawKey`（リンク）」と「パスフレーズ（別経路）」の**両方**が揃わないと再現できない。
- `salt` は KV に保存してよい（パスフレーズ自体ではなく派生の材料であり、単体では無価値）。

### 4.2 URL 構造

```
https://<host>/s/#<id>.<key>
                 └─┬─┘ └─┬─┘
                   │      └ base64url(rawKey 32B) … 復号鍵。サーバーに送らない。
                   └ base64url(9B 乱数) = 12文字 … 暗号文の lookup handle。
```

- `id` も**フラグメントに置く**理由: `/s/` のページロード要求とそのアクセスログに、識別子すら残さないため。
  `id` はページ内 JS が抽出し、`GET /api/secret/:id` で初めて送信する。
- ブラウザは `#` 以降を HTTP 要求行（request-target）に含めない（RFC 3986 / フラグメントはクライアント側で解釈）。
  この仕様を鍵運搬に流用しているのが本設計の肝。

### 4.3 認証付き暗号（改竄検知）

AES-GCM は復号時に認証タグを検証する。暗号文を 1 bit でも改竄すると復号が失敗する。
そのため「すり替えられた暗号文を気づかず読む」が原理的に起きない。
**「パスフレーズ不一致」と「改竄」は GCM では区別できず、いずれもタグ検証失敗として同一エラーになる。**

## 5. 脅威モデル

### 5.1 守れること（防御対象）

| 脅威 | なぜ守れるか |
|---|---|
| KV / DB の流出 | 保存しているのは鍵なしの AES-256 暗号文。総当たりは計算量的に不可能（2²⁵⁶）。 |
| サーバーログ流出 | 鍵もページURLフラグメントもサーバーに届かないため、ログに現れない。 |
| 運営者の受動的閲覧 | 平文も鍵も保持しないため、覗いても暗号文しか見えない。 |
| 暗号文の改竄 | GCM 認証タグ検証で復号が失敗する。 |
| リンクの再利用・遅延傍受 | ワンタイム破棄により、開封後は取得対象が存在しない（露出窓最小化）。 |

### 5.2 検知できること

- 受信者が開く前に既に burn 済みなら、「**誰かが先に開いた＝傍受**」のサイン（トリップワイヤー）。
  → 運用上は受信者に「すぐ開く／開けなければ連絡」を依頼し、検知時は当該秘密をローテートする。

### 5.3 守れないこと（範囲外・要明記）

| 脅威 | 状況 | 緩和策 |
|---|---|---|
| リンク送信経路の漏洩 | リンク（# 含む）を見た者は復号可能 | パスフレーズを別経路で渡す（§9） |
| 悪意ある運営者（能動） | 復号 JS をサーバーが配る以上、バックドア版を配れば鍵・平文を抜ける（ブラウザ E2E の根本限界） | OSS 化＋SRI、究極はブラウザ拡張／ネイティブ配布 |
| メタデータ | `id`・時刻・IP/国・暗号文サイズはサーバーから見える（中身ではない） | 最小限の収集に留める |
| 端末汚染 | 復号後の平文は受信端末上に存在する | 方式によらず防御不能（運用で対処） |

> 設計思想: 「サーバーを信頼しないで済む」状態にすること。最も漏れやすい**保管中の平文**を消し、
> 残りの露出窓（経路・存在時間・再利用）を burn / TTL / パスフレーズで順に塞ぐ。

## 6. データモデル（KV）

| 項目 | 値 |
|---|---|
| キー | `s:<id>` |
| 値（JSON） | `{ ct, iv, salt, maxViews, views, expiresAt }` |
| ct | base64url（暗号文＋GCMタグ） |
| iv | base64url（12B） |
| salt | base64url（16B）または `null`（パスフレーズなし） |
| maxViews | 整数（既定 1） |
| views | 整数（初期 0、読むたび +1） |
| expiresAt | 絶対エポック秒（TTL の真実の源） |

**TTL の扱い（注意点）**: Workers KV の `put` は既定で TTL を引き継がない。
`maxViews > 1` で読み取り後に再 `put` する場合、`expiration`（絶対エポック）を**毎回明示**して
残存 TTL を維持する。値内の `expiresAt` を基準に `put(..., { expiration: expiresAt })` とする。
（`maxViews = 1` なら初回読み取りで delete するため再 put は発生しない。）

## 7. API 仕様

ベースパス `/api`。すべて JSON。CORS は同一オリジン配信なら不要、別オリジンなら許可。

### 7.1 `POST /api/secret` — 封緘（暗号文の保管）

リクエスト:
```json
{ "ct": "...", "iv": "...", "salt": "..."|null, "maxViews": 1, "ttl": "1h"|"24h"|"7d" }
```
レスポンス（201）:
```json
{ "id": "AbC123dEf456" }
```
- バリデーション失敗時は `400 { "error": "bad request" }`。
- サーバーは鍵・平文を受け取らない。`ttl` は `expiresAt` に変換して保存。

### 7.2 `GET /api/secret/:id` — 開封（取得＋ワンタイム破棄）

レスポンス（200）:
```json
{ "ct": "...", "iv": "...", "salt": "..."|null, "maxViews": 1, "views": 1 }
```
処理:
1. `s:<id>` を取得。無ければ `404 { "error": "gone" }`。
2. `views += 1`。
3. `views >= maxViews` → `delete`（burn）。そうでなければ `expiresAt` 付きで再 `put`。
4. 暗号文を返す（復号はクライアント側）。

- 期限切れは KV の TTL 失効により自然に `404`。
- レスポンスに鍵は含まれない（クライアントが `#` から保持）。

## 8. フロントエンド設計

### 8.1 画面状態（state machine）

```
create ──seal──▶ sealed ──"open as recipient"/リンク到達──▶ recipient
                                                              │ reveal
                                                              ▼
                                                  revealed │ error(gone/expired/badkey)
```

- `create`: 秘密入力、`EXPIRES`（1h/24h/7d）、`OPENS`（once/3×/10×）、パスフレーズ任意。
- `sealed`: 生成リンク表示＋コピー。「中身は見ていない」明示。
- `recipient`: 「開くと破棄」警告。パスフレーズ必要時は入力欄。
- `revealed`: 平文表示＋コピー。以後リンクは無効。
- `error`: `gone`（開封済/不在）/ `expired` / `badkey`（パスフレーズ違い or 改竄）。

### 8.2 クライアント処理

- 封緘: 鍵生成 → 暗号化 → `POST /api/secret` → `#<id>.<key>` を組み立ててリンク提示。
- 開封: `location.hash` を `.` で分割し `id` と `key` を取得 → `GET /api/secret/:id` → `key`（＋パスフレーズ）で復号。
- 鍵は常にメモリ内のみ。`localStorage` 等へ保存しない。

## 9. 受け渡し運用（Handoff）

| 機微度 | リンク | パスフレーズ | 補足 |
|---|---|---|---|
| 通常 | 既存チャット/メール | 不要 | 平文残置を消すだけで大きな改善（恒久性・検知性・影響範囲の差） |
| 機微 | 経路A（Slack） | 経路B（SMS/口頭/別アプリ） | 独立2経路を要求。本来のパスフレーズの用途 |
| 最高 / 対面 | 画面QRを読み取り | 口頭 | 長URLの読み上げをQRで回避 |

運用ルール: 受信者へ「**すぐ開く／"already opened" が出たら連絡**」を依頼。検知時は当該秘密を即ローテート。

## 10. 「正確に一度」の保証（KV と Durable Object）

- **KV は結果整合**。複数リージョンで同時に開くと、delete 伝播前に両方が読めてしまう＝**ベストエフォートのワンタイム**。
- 厳密な「exactly once」が必要なら、`id` を **Durable Object**（世界で一意・単一スレッド）にルーティングし、
  read→burn を直列化する。DO 内で `get` → `views++` →（閾値到達なら）`deleteAll` を原子的に行う。

判断指針: 安価・グローバル・ベストエフォートで足りるなら KV。
「絶対に一度きり」を約束するなら DO。**この選択自体が設計判断**であり、要件（機微度）で決める。

---

## 11. 運用・コスト・拡張

- **コスト/運用**: サーバーレス、330+ ロケーション配信、無料枠で広範に賄える。スケール時もインフラ管理なし。
- **可観測性**: 収集は最小限（封緘数・開封数・失効数・国別程度）。**中身・鍵は記録しない**。
- **拡張**:
  - 開封通知: 開封イベントを Queues 経由で メール/LINE に送る（トリップワイヤーの自動化）。
  - ファイル秘密: KV 値上限（~25MiB）超は R2（下り課金ゼロ）に暗号文を保管。
  - 監査ライト: `request.cf.country` 等の最小メタのみ記録、内容は不可。

## 12. 受け入れ基準（テスト観点）

| # | シナリオ | 期待結果 |
|---|---|---|
| 1 | 封緘 → 開封 | 平文が復号できる |
| 2 | 同一リンクで2回目の開封（maxViews=1） | `gone`（取得対象なし） |
| 3 | 暗号文を改竄して開封 | `badkey`（GCM 検証失敗で復号不可） |
| 4 | パスフレーズ必須を誤入力 | `badkey` |
| 5 | TTL 経過後の開封 | `gone`（KV 失効） |
| 6 | maxViews=3 | 3回開封可、以後 `gone`、各回で TTL 維持 |
| 7 | ネットワーク監査 | `#` 以降（鍵）がサーバー宛リクエストに一切現れない |
| 8 | KV ダンプ検査 | 鍵なしでは平文復元不可（暗号文・iv・salt のみ） |

## 13. 設計判断の記録（なぜ）

- **鍵を URL フラグメントに置く**: `#` 以降は仕様上サーバーに送られない。これを鍵運搬に流用し、サーバーを信頼対象から外す。
- **AES-256-GCM**: 機密性に加え認証タグで改竄検知。すり替え暗号文を気づかず復号する事故を防ぐ。
- **PBKDF2 でパスフレーズ派生**: 「リンク」と「パスフレーズ」を独立2要素にし、経路漏洩への耐性を付与。反復回数で総当たりを律速。
- **KV ＋（必要時）DO**: 露出窓は TTL で縮め、ワンタイム厳密性は DO で担保。コストと保証のトレードオフを要件で選ぶ。
- **ワンタイム破棄**: 露出窓の最小化に加え、傍受の検知（トリップワイヤー）という副次効果を得る。

---

## 14. 既知の制約・リスク

- 復号 JS の配信はサーバーに依存（§5.3）。OSS/SRI で緩和するが、能動的悪意は範囲外。
- リンク経路が漏れれば（パスフレーズなしの場合）復号され得る。機微時はパスフレーズ必須運用とする。
- KV のみではワンタイムはベストエフォート。厳密版は DO を採用。
- メタデータ（時刻・IP/国・サイズ）は不可避に観測され得る。中身ではないが収集は最小化する。
